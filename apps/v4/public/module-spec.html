MODULE 01

Trips / Dispatch

Data Captured
  • public_id — human-readable trip ref
  • status — created / dispatched / started / completed / canceled
  • scheduled_at & estimated_end_date (ISO datetime)
  • pickup_uuid, pickup_name, pickup_code (Place)
  • dropoff_uuid, dropoff_name, dropoff_code (Place)
  • driver_assigned_uuid & driver_assigned (Driver object)
  • vehicle_assigned_uuid & plate_number
  • fleet_uuid & fleet_name
  • block_id (groups multi-stop rows in CSV import)
Key Entities

Order/Trip Driver Vehicle Place Fleet

Statuses: createddispatchedstartedcompletedcanceled

Main Screens / Routes
  • /trips — AG Grid list, status badges, KPI cards
  • /trips?status=dispatched — server-side status filter
  • /trips?tab=history — historical date-range view
  • Slide-over panel — New Trip form (inline)
  • Filter panel (right drawer) — status, driver, fleet, pickup, dropoff, date range
  • Import wizard (CSV/Excel) — 2-step: create places → import orders
Core Workflow
1
Dispatcher creates trip (or bulk-imports CSV). Status = created.
2
Dispatcher assigns driver via dropdown (shows rota availability: WD/RD/HOL badges). Assigns vehicle.
3
Dispatcher clicks Dispatch → status = dispatched. Driver receives push notification.
4
Driver taps "Start" on mobile → status = started.
5
Driver taps "Complete" → status = completed. POD/signature captured.
Alerts / Notifications
  • Trip dispatched → push to driver
  • Unassigned trips → amber KPI card warning
  • Driver blocked (RD/HOL/UNAVAILABLE) → blocked in dropdown, cannot be selected
  • Driver outside preference window → amber ⚠ warning in dropdown
Reports / Exports
  • CSV export of current filtered grid view
  • Failed rows file download on import errors
  • Filters: by status, driver, fleet, pickup/dropoff text, date range
Unique Behaviours
  • Driver dropdown cross-checks rota store — shows WD (working day with start time), RD, HOL, N/A, ? badges in real-time
  • Blocked drivers are disabled (cannot be selected even if clicked)
  • CSV import auto-creates missing Places before importing Orders (2-phase)
  • AG Grid: 15 rows/page, column sort/reorder/resize, client-side search across all columns
  • History tab enforces date range — rejects future dates, requires from <= to
MODULE 02

Driver App

Screens in Order
  • Login — email/password, company scoping
  • Today's Trips — list of dispatched trips for today
  • Trip Detail — pickup/dropoff addresses, scheduled time, vehicle plate
  • Walkaround Check — full checklist before first trip of day
  • Start Trip — tap to begin, GPS timestamp recorded
  • En Route — map/status screen during trip
  • Complete Trip — POD capture, e-signature, optional photo
  • Defect Report — report issue found mid-trip or post-trip
  • Driver Hours — view today's hours, WTD summary
  • Profile — name, licence, CPC hours summary
Driver Actions (UI interactions)
  • Tap to confirm walkaround item (OK / Advisory / Fail)
  • Camera tap to capture photo evidence
  • Type full name to e-sign walkaround / PMI
  • Tap Start / Complete on trip card
  • Form fill for defect description
  • GPS auto-captures location on submit
Data Driver Provides
  • Walkaround answers (ok/advisory/fail per item)
  • Defect descriptions & severity
  • Photo uploads (JPEG/PNG)
  • E-signature (typed name)
  • GPS coordinates (auto)
  • Trip start/end timestamps (auto)
Driver Notifications Received
  • New trip dispatched → push notification
  • Schedule changes → push notification
  • Defect status updates → push notification
  • Compliance reminders (CPC, licence expiry) → push/email
MODULE 03

Rota & Scheduling

Data Captured
  • driver_uuid, date (YYYY-MM-DD)
  • status — WD / RD / HOL_REQ / UNAVAILABLE / OFF
  • trip_uuids[] — trips assigned on a WD cell
  • note — optional free-text per cell
  • preference_start / preference_end — "HH:MM" shift preference window
  • leave_requests — start_date, end_date, status, user_uuid
Main Screens / Routes
  • /rota — 7-column weekly grid (Mon–Sun), one row per driver
  • Cell popover — status picker + trip assignment per driver/day
  • Trips Dock (right sidebar) — unassigned trips for the week, draggable
  • Preference Editor (inline cell) — HH:MM start/end window per driver
Core Workflow
1
Animated loader fetches drivers, leave requests, rota entries, trips in parallel.
2
Dispatcher clicks a cell → popover: set WD/RD/HOL/UNAVAILABLE/OFF + optional note.
3
On WD: see unassigned trips for that date, tick to assign. Or drag trip card from dock onto cell.
4
Save → API updates trip's driver_assigned_uuid. Dock auto-refreshes.
5
Week navigation via < > arrows. ISO week number displayed.
WTD / Hours Logic
  • HOL_REQ status sourced from Leave Requests API
  • Leave matched by user_uuid (primary) or driver name (fallback)
  • Preference window shown in Trips Dispatch dropdown — ⚠ if trip time outside window
  • No automatic WTD calculation in rota (tracked separately in Driver Hours module)
  • Blocked statuses (RD/HOL_REQ/UNAVAILABLE/OFF) prevent assignment in Trips page dropdown
Unique Behaviours
  • Drag-and-drop: trip cards from dock → rota cell. Conflict dialog if cell already has trips.
  • Status stored in browser localStorage (rota-store) — lightweight, fast, offline-capable
  • 4-step animated loader with progress bar and skeleton grid preview while API loads
  • Pending drivers are filtered out (status !== "pending")
  • Duplicate drivers deduped by uuid AND by name+phone
MODULE 04

DVSA Walkaround Checks

Data Captured
  • vehicle_uuid, driver_uuid, company_uuid
  • check_date, check_time (ISO datetime)
  • elapsed_seconds (duration timer)
  • gps_lat, gps_lon, location_name
  • anti_cheat_photo (random prompt: nearside rear wheel, tread depth, brake lamp, etc.)
  • Per item: item_id, answer (ok/advisory/fail/yes/no/text/number), photo_url
  • defects[]: category, item_name, defect_type (Advisory/Dangerous), notes, photo_url
  • signature_name, declaration_text
  • template_id (which checklist template was used)
Standard Checklist Sections
  • Tyres & Wheels (5 items: tread/condition/pressure, dual rear, wheel nuts)
  • Lights (6 items: headlights, tail, brake, indicators, reverse+audible, markers)
  • Brakes (3 items: air pressure build-up, handbrake, pedal feel)
  • Fluids & Engine (5 items: oil, coolant, AdBlue, leaks, washers)
  • Body & Cab (5 items: windscreen, wipers, mirrors, horn, body damage)
Question Types

ok_advisory_fail pass_fail yes_no text number select

Photo requirements per item: none optional required required_on_fail

Defect severity: advisory dangerous

Main Screens / Routes
  • /compliance → Walkaround tab — KPI cards, recent checks list
  • New Check form — vehicle/driver selectors, live timer, anti-cheat prompt, checklist
  • Check detail view — summary card, defect callout, full section grid, PDF download
  • Walkaround Templates tab — create/edit custom templates with conditional logic
Sign-Off Flow
1
Driver selects vehicle (auto-selects matching template) and driver name.
2
Anti-cheat random photo prompt shown — must photograph specific part.
3
Complete all required items; mandatory photos on fails must be captured.
4
Type full name into signature field → click Sign.
5
Submit button unlocks. On submit: defects → workshop alert + VOR flag if dangerous.
Alerts
  • Defect reported → push + email to transport manager and workshop
  • Dangerous defect → VOR flag raised automatically (vehicle taken off road)
  • Check completed <2 mins with >50% answers → "suspiciously fast" manager alert
  • Vehicle pending check → shown as pending count in KPI cards
Reports / Exports
  • PDF report per check (via API)
  • List view: all checks, filterable by date/driver/vehicle/status
  • KPIs: checks today, defects today, average check time, pending vehicles
Unique Behaviours
  • Template auto-selected based on vehicle plate number pattern matching
  • Conditional items: items can appear/hide based on parent item answer
  • Progress bar tracks answered/total; submit blocked until all required + photos done
  • GPS coordinates embedded in signed declaration (lat/lon + location name)
  • API limit: driver list capped at 100 (500 causes 500 server error)
MODULE 05

Defect Management

Data Captured
  • defect_id, date, vehicle_reg
  • reporter_name, reporter_role (Driver / Technician)
  • source — Walkaround / PMI / Manual
  • item (defect category) — Warning Light / Tyre / Brake / ABS
  • description (free text)
  • severity — advisory / dangerous
  • status — open / resolved
  • parts_used, labour_hours (on rectification)
  • signed_off_by, signed_off_date
  • photo_url[]
Defect Lifecycle
1
Reported — driver marks fail on walkaround OR technician flags on PMI.
2
Triaged — workshop receives alert. Severity confirmed (Advisory vs Dangerous).
3
In Progress — technician logs parts + labour in rectification form.
4
Resolved — technician clicks "Mark Roadworthy & Sign Off". Status = resolved.
5
Archived — full audit trail retained with who signed off and when.
Main Screens
  • /maintenance → Defects tab — list view, filter open/resolved
  • Defect card — expandable with rectification record, sign-off form
  • KPI cards: Open Defects, Resolved, Total
Alerts
  • Any defect reported → email + push to transport manager + workshop
  • Dangerous defect → VOR raised, vehicle grounded immediately
Unique Behaviours
  • Dangerous defects auto-lock vehicle (VOR) — cannot be dispatched
  • Defect thread shows parts used, labour time, sign-off name + date
  • Defects from walkaround and PMI are unified in same defect log
MODULE 06

Vehicle Maintenance (PMI)

Data Captured
  • vehicle_id, reg, make, model, year, weight, VIN
  • mot_expiry, tacho_expiry, loler_expiry, loler_type
  • pmi_interval_weeks (4 / 6 / 8 weeks)
  • last_pmi_date → auto-calculates next_pmi_date
  • status — green (compliant) / amber (due soon) / red (overdue/VOR)
  • Per PMI item: pass / advisory / fail + photo + severity
  • Brake test: axle 1 efficiency %, axle 2 efficiency %
  • technician_name, signature_name, submission_timestamp
PMI Checklist Sections
  • Brakes (5 items)
  • Steering (3 items)
  • Tyres & Wheels (4 items)
  • Lights & Electrics (6 items)
  • Body & Cab (5 items)
  • Engine & Fluids (5 items)
  • Suspension & Chassis (4 items)
  • Trailer Coupling (4 items)
  • Brake Test (mandatory DVSA — axle efficiency %)
Main Screens / Routes
  • /maintenance → Dashboard — VOR count, amber, green KPIs, Earned Recognition, 8-week schedule
  • /maintenance → PMI Sheet — sorted vehicle list, urgency order
  • PMI Detail Sheet — 3-col checklist grid, brake test, technician declaration, e-sign
  • /maintenance → Defects — unified defect log
  • /maintenance → Settings — vehicle profiles, user roles, notifications, custom checklist items
Earned Recognition KPIs
  • PMI Completed On Time % (target: 100%)
  • Defects Found & Fixed % (target: 100%)
  • Roadworthiness Rate % (target: 100%)
  • DVSA Scheme badge displayed prominently
Alerts / Notifications
  • 7 days before PMI due → email + push to manager
  • PMI fail → SMS + email to manager immediately
  • Vehicle VOR → push to manager (optional toggle)
Reports / Exports
  • Full Vehicle History PDF (all PMI reports per vehicle)
  • Compliance Report (DVSA Earned Recognition format)
  • Defect Log CSV
  • Samsara / Webfleet live mileage API integration capability
Unique Behaviours
  • PMI interval is per-vehicle (4/6/8 weeks) not global
  • Next PMI date auto-calculated from lastPMI + interval_weeks × 7
  • Brake efficiency <50% → DVSA FAIL shown in red automatically
  • Traffic light board shows red → amber → green sorted order
  • 8-week rolling forward schedule groups vehicles by week due
  • Custom checklist items configurable per depot (tail lift, HIAB, reefer)
MODULE 07

Driver Hours / WTD

Data Captured (Compliance → Drivers tab)
  • tacho_card_ref, tacho_card_expiry
  • dqc_number, dqc_expiry (Driver Qualification Card)
  • cpc_cycle_start, cpc_cycle_end, cpc_hours (of 35), cpc_deadline
  • cpc_modules[]: date, subject, provider, atp_ref, hours
  • davis_risk_score (0–100), risk level (low/medium/high)
  • recheck_frequency (monthly if 9+ pts, quarterly if 3–8, 6-monthly if clean)
  • licence_points, endorsements[], entitlements[] (categories + expiry)
  • leave_requests bridged from Rota & Scheduling
WTD Limits Tracked
  • CPC: 35 hours per 5-year cycle (segmented bar: 5 × 7h blocks)
  • Tachograph card expiry monitoring
  • DQC expiry monitoring
  • DVLA licence check frequency: monthly (9+ pts), quarterly, 6-monthly
  • ADR certificate expiry
  • Medical D4 form tracking
  • Right to Work / visa expiry
Alerts Triggered
  • 9+ points → automatic monthly DVLA check, HR notified
  • CPC deadline <90 days → KPI "CPC Critical" badge
  • Licence check due <30 days → "Checks Due" KPI
  • Right to Work expiring <90 days → "RTW Expiring" KPI
  • Driver disqualified → "Disqualified" red KPI, must not drive
  • Tacho card expiring → compliance dashboard amber/red
Main Screens
  • /compliance → Drivers tab — driver list with risk scores
  • Driver detail: Licence · CPC · Training · Right to Work · Audit Trail sub-tabs
  • CPC tab: segmented 5-block progress bar, module cards, outstanding placeholders
  • Audit Trail: DVLA check history table, CSV export
  • /compliance/driver-hours-wtr — dedicated WTD overview page
Unique Behaviours
  • DAVIS risk score drives automated recheck frequency (not manual)
  • Licence numbers partially masked (last 4 chars shown) for data protection
  • GDPR consent date recorded; audit trail only visible after written consent
  • Previous CPC cycle toggle shows historical 35h completion
  • Right to Work: EU settled status vs Skilled Worker visa vs UK citizen tracked differently
MODULE 08

Toll Tracking

Data Captured
  • toll_id (e.g. TE-2026-041)
  • date, vehicle_reg, driver_name
  • route — e.g. "M6 Toll – Junction 11–3A", "Dartford Crossing (ANPR)"
  • amount (£GBP), toll_type — Motorway / Bridge / Tunnel
  • payment_method — OBU (On-Board Unit) / ANPR / Cash
  • status — reconciled / pending
Toll Routes Covered
  • M6 Toll (Junction 7–3A, 11–3A) — £11.00 HGV
  • Dartford Crossing (ANPR) — £6.00
  • Humber Bridge — £3.00
  • A19 Tyne Tunnel — £4.60
  • Mersey Gateway Bridge — £4.50
  • A14 Cambridge Northern Bypass — £5.80
Main Screens / Routes
  • /toll-expenses — filterable table: all tolls
  • /toll-receipts — receipt-level view
  • KPI cards: Total Cost, This Month, Pending, OBU count
  • Filter: all / reconciled / pending + search by reg/driver/route
Attribution
  • Per vehicle (reg plate)
  • Per driver (name)
  • Per trip (linked via date + vehicle)
  • Status: pending → reconciled (manual action)
Reports / Exports
  • CSV export of filtered view
  • Total cost summary (all + filtered subset)
  • Pending reconciliation count
Unique Behaviours
  • OBU charges imported automatically; Cash/ANPR require manual entry
  • Toll amount is NOT auto-calculated from trip route (manually entered or imported)
  • Reconcile button shown on pending rows; View shown on reconciled
  • Type colour coding: Motorway=blue, Bridge=indigo, Tunnel=purple
MODULE 09

Amazon AFP (Amazon Freight Partner)

AFP-Specific Fields
  • block_id — Amazon's grouping key for multi-stop routes in CSV import
  • pickup_code / dropoff_code — Amazon warehouse codes (e.g. BHX1, MAN2)
  • fleet_name — linked to Amazon fleet contract
  • driver_name — matched to Amazon-registered driver list
  • plate_number — must match vehicle registered with Amazon
  • scheduled_at / estimated_end_date — Amazon dispatch windows
Amazon-Required Compliance
  • DVSA walkaround check must be completed before first Amazon block
  • Driver CPC must be current (35h complete)
  • Vehicle MOT must be valid
  • Right to Work verified for all non-UK drivers
  • Tachograph card must not be expired
AFP Import Format

CSV columns required:

  • block_id
  • driver_name
  • plate_number
  • scheduled_at
  • estimated_end_date
  • pickup_code
  • dropoff_code
  • fleet_name
Import Process
1
Upload Amazon schedule CSV (from Relay/AFP portal).
2
System auto-creates missing Places from pickup_code/dropoff_code.
3
Trips created, grouped by block_id.
4
Failed rows available as downloadable error log CSV.
Reports Amazon Requires
  • Walkaround check records (per trip, per vehicle)
  • Driver compliance summary (CPC, licence, medical)
  • Vehicle MOT & PMI records
  • Incident / defect reports
Unique Behaviours
  • block_id groups multiple CSV rows into a single trip (multi-stop)
  • Amazon session authentication checked before importing (login session validation)
  • Driver compliance gate: system warns if compliance docs missing before AFP block dispatch
MODULE 10

Multi-lingual

Supported Languages

EN — English PL — Polish RO — Romanian

Switch via top bar language selector. Selection persists in localStorage.

Translation Architecture
  • Centralised useLang() hook — returns t translation object
  • All UI strings accessed via t.module.key (e.g. t.rota.workingDay)
  • Translation object structured by module: common, trips, rota, compliance, maintenance, etc.
  • LangContext Provider wraps entire app
What IS Translated
  • Navigation labels, page titles, tab names
  • Button text, status labels, filter labels
  • Rota status names (Working Day / Dzień Roboczy / Zi Lucrătoare)
  • Common UI strings (Search, Export, Save, Cancel, etc.)
  • KPI card labels, form field labels
What is NOT Translated
  • Regulatory text (DVSA declarations — legally must remain in English)
  • Vehicle registration plates
  • Place names (depot codes, Amazon warehouse codes)
  • Driver/vehicle data from API responses
  • Error messages from API
  • PDF reports (generated in English)
Switch Mechanism
  • Globe icon in top bar triggers language dropdown
  • Instant re-render — no page reload required
  • Language code stored: "en" / "pl" / "ro"
MODULE 11

Compliance Dashboard

Dashboard Tabs
  • Overview — aggregate KPIs: high risk drivers, CPC critical, checks due, RTW expiring, disqualified count
  • Documents — compliance matrix: O-Licence, Vehicle, Trailer, Driver, Workshop, Insurance, Audit, H&S documents
  • Drivers — per-driver: licence, CPC, training, RTW, DVSA audit trail
  • Vehicles — MOT, tachograph, LOLER, speed limiter tracking
  • Training — CPC modules, ADR, HIAB, manual handling, drug & alcohol
  • Audit — DVSA encounters, PG9 prohibitions, FORS cert, earned recognition
  • Settings — compliance manager, notification channels, expiry alerts, event alerts, integrations
RAG Status Logic

🔴 RED (≤30 days or expired)

🟡 AMBER (31–90 days)

🟢 GREEN (>90 days)

Applied to: MOT, tacho expiry, LOLER, CPC deadline, DQC, licence check date, RTW/visa expiry, ADR cert, training records, all insurance policies.

Document Compliance Matrix
  • Rows: document names (54 total across 8 categories)
  • Columns: entity (vehicle reg or driver name) or "All"
  • Cells: expiry date → RAG badge, two signature slots, file attachments
  • Upload/download documents per cell
Notification / Alert Engine

Expiry Alerts — multi-phase:

  • Early warning: 90d or 60d (configurable) → Compliance Manager first
  • Reminder: 30d / 21d / 14d (configurable)
  • Critical: always 7d → All admins

Event Alerts — real-time, instant:

  • Defect reported
  • Vehicle VOR
  • Licence points added
  • Driver disqualified
  • Document uploaded/signed
Alert Configuration
  • Compliance Manager selectable from user list
  • Email channel: on/off + test send
  • Mobile push: on/off + test send
  • Daily digest: batch non-urgent into one morning email (configurable time)
  • Per-alert type: email + mobile toggles independently controlled
  • Escalation: non-urgent → CM first; operational → all admins always
Document Categories Tracked

O-Licence Vehicle MOT Tacho Cal LOLER Trailer MOT Driver RTW Licence Checks ADR Insurance FORS DVSA Earned Recognition H&S Risk Assessments Workshop Records

Integrations (Settings → Integrations)
  • DAVIS API — DVLA licence check integration
  • Samsara — live vehicle telematics
  • Webfleet — live mileage / GPS
  • Amazon AFP — relay schedule sync
MODULE 12

User Roles

Driver

Mobile app only (dedicated driver app)

  • View own trips (today + upcoming)
  • Complete walkaround check
  • Start / complete trips with GPS
  • Report defects
  • View own CPC hours, licence status
  • Cannot access dispatcher or compliance data
  • Cannot see other drivers' data
Dispatcher

Web app — operational focus

  • Create, edit, dispatch, cancel trips
  • Assign drivers and vehicles to trips
  • Manage rota (set WD/RD/HOL etc.)
  • View walkaround check status
  • View toll expenses
  • Cannot access compliance settings or licence data
  • Cannot manage users or system configuration
Transport Manager / Admin

Web app — full access

  • All dispatcher permissions
  • Full compliance dashboard
  • Driver compliance records (licence, CPC, RTW)
  • Vehicle maintenance (PMI, defects)
  • Document management (upload, sign off)
  • Configure notification settings
  • Export all reports
  • Manage user roles
Technician

Web app — maintenance only

  • Access PMI sheets only
  • Complete and sign off PMI inspections
  • Log and resolve defects
  • Cannot access trip dispatch or driver compliance data
Auditor / DVSA Inspector

Web app — read-only

  • Read-only access to all compliance records
  • View walkaround check history
  • View PMI records and defects
  • View document matrix
  • Cannot edit, create or delete any records
Company Scoping
  • All data is scoped to company_uuid — multi-tenant
  • Drivers filtered by company_uuid in walkaround check form
  • API endpoints require authenticated session token
  • JWT-based authentication (Supabase / OnTrack API)
  • Vehicle and driver lists never bleed across company boundaries
