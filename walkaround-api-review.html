Critical Critical Issues

1 Login endpoint uses wrong HTTP method

GET {{url}}/int/v1/auth/login

A login that sends credentials in a JSON request body must use POST, not GET. A GET with a body is non-standard — many proxies strip it, credentials leak into server access logs and browser history, and it violates HTTP semantics (GET should be safe & idempotent).

- GET {{url}}/int/v1/auth/login
+ POST {{url}}/int/v1/auth/login

2 Missing authentication mechanism documentation

The docs state "All endpoints require a valid session cookie / authenticated user" but the login response returns a token field. There is no explanation of how that token should be used:

  • Is it sent as Authorization: Bearer {{token}}?
  • Is it set as a cookie automatically by the server?
  • What is the token lifetime / refresh strategy?

This is a critical omission — consumers cannot integrate without this information.

3 responses[].photo_uuid marked as unconditionally required

POST /walkaround-checks — Submit Check (Endpoint 11)

The field reference marks photo_uuid as "required" for every response. However, the photo_required_on_fail flag on categories implies photos are only conditionally required. The error message "photo_uuid is required for category '...'" also implies selective enforcement.

Recommendation: Change to "Required if photo_required_on_fail = true for the category and the response is Fail, or always required if configured — clarify business rule."

Significant Significant Issues

4 Anticheat category endpoint — no date filtering

GET /walkaround-templates/{id}/anticheat-category (Endpoint 7)

The anti-cheat category is described as "deterministically rotated daily" but there's no date query parameter. Consumers can only get today's category — making it impossible to audit past rotations or run regression tests for specific dates.

5 defect_type field values overlap with response values

POST /walkaround-checks — Submit Check (Endpoint 11)

The response field accepts OK, Advisory, or Fail. The defect_type field accepts Advisory or Dangerous. The value "Advisory" is used in both fields with different meanings, which is confusing:

  • response: "Advisory" — a non-failing notice?
  • defect_type: "Advisory" — a severity classification on a Fail?

The documentation should clearly explain the semantic difference and consider renaming one to avoid ambiguity (e.g. defect_severity for the defect classification).

6 No endpoint to unassign vehicles from a template

POST /walkaround-templates/{id}/assign-vehicles (Endpoint 6)

There is an endpoint to assign vehicles but no endpoint to unassign a vehicle from a template entirely (without reassigning to another). The only way is via the vehicle_ids array in PUT /walkaround-templates/{id}, which is a full replacement. This should be explicitly documented, or a DELETE .../unassign-vehicles endpoint should be provided.

7 Missing page query parameter on paginated endpoints

All list endpoints (1, 9, 13)

All paginated endpoints document a limit parameter and return meta.current_page / meta.last_page, but none document a page query parameter. How does a consumer navigate to page 2 and beyond?

Minor Minor Issues & Inconsistencies

8 Route conflict: /vehicle-map vs /{id}

GET /walkaround-templates/vehicle-map (Endpoint 13)

If the router processes the {id} wildcard path before the literal vehicle-map path, "vehicle-map" will be interpreted as an ID and likely return a 404. Route registration order must be verified — literal routes registered before wildcard routes.

9 Inconsistent summary fields across list responses

GET /walkaround-checks (Endpoint 9) vs GET /walkaround-templates/vehicle-map (Endpoint 13)

The List Checks response includes avg_check_time_formatted: "6m 30s", while the Vehicle Map response includes avg_check_time_minutes: 6.5. These represent the same data but use different field names and formats. Standardize to include both in both responses, or use consistent naming.

10 ID field naming inconsistency: *_id vs *_uuid

POST /walkaround-checks — Submit Check (Endpoint 11)

The request uses driver_id, template_id, vehicle_id (accept UUID or public ID), but responses use driver.uuid. The documents should consistently clarify that *_id fields accept either format, ideally with a note in a common conventions section.

11 signature field — no retrieval mechanism

POST /walkaround-checks — Submit Check (Endpoint 11)

The signature field is documented as "File UUID of the signature image" in the request, and the response returns a raw UUID string. However, there's no URL or documented endpoint for consumers to retrieve and render the actual signature image from this UUID.

12 Missing 401 / 403 error responses

No endpoint documents 401 Unauthorized or 403 Forbidden responses, despite the header stating authentication is required. The docs should include these at a minimum in a global "Common Error Responses" section — covering expired tokens, missing auth headers, and cross-company access attempts.

13 No documented max for vehicle_ids array

POST /walkaround-templates — Create Template (Endpoint 3)

The field reference documents limits for items (max 50) and items[].categories (max 100), but vehicle_ids has no stated limit. If unbounded, this could be a performance or abuse vector.

14 is_master flag — undocumented

The is_master field appears in template responses but is never defined in any field reference. What triggers it? What restrictions does it impose? The update endpoint mentions "only company-owned templates can be updated (not master templates)" — this logic and the flag itself should be explicitly documented.

15 record_status in category response — undocumented

GET /walkaround-checks/{id} — Get Check (Endpoint 10)

The response includes "record_status": 1 in category objects but this field is never documented. What are the possible values, and what do they mean? (e.g. 1 = active, 0 = soft-deleted?)
