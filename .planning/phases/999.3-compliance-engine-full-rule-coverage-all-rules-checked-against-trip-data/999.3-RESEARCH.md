# Phase 999.3: Research — Compliance Engine Full Rule Coverage

## Summary

- **Critical Bug (D-10):** `assimilated.ts` L134 guards per-day rules with `if (!day.hasDriving) continue`. Since ALL current trips are `NON_DRIVING_DUTY`, `hasDriving` is always `false` — the daily 13h/15h working hours limit **never fires** in the batch validator for any real trip.
- **Block Removal (D-04):** `prospective-check.ts` L192–203 blocks assignment with `MISSING_END_TIME` when a trip has no end time. This must be removed — the 2h fallback in `tripEnd()` (L74–75) is sufficient for gap checking.
- **UI Badge (D-05):** "Est. End" column in `trips/page.tsx` (`colDefs` L1730–1737) renders the `estimated_end_date` field. This cellRenderer must be updated to show an amber "No end time" badge when both `estimated_end_date` is null/empty AND `order.time` is 0/null.
- **Test Gaps:** Weekly rest (45h/24h), fortnightly driving (90h), reduced rest limit (3×/week cap), daily working hours for NON_DRIVING_DUTY days (after D-10 fix), and clean-slate driver (no data = no violations) — none of these are currently tested.
- **Flag:** `breaksAssumedCompliant: boolean` goes in `VehicleConfig` (types.ts L167–173). Wire into `assimilated.ts` break check — when `true` (default), break check only fires for `DRIVING` days (current behaviour made explicit).

## Critical Bug Analysis

### D-10: The hasDriving Guard Bug

**File:** `apps/v4/lib/compliance-engine/assimilated.ts`
**Current code (L133–136):**
```typescript
for (const day of days) {
  if (day.isRestDay) continue
  if (!day.hasDriving) continue   // ← BUG: skips all NON_DRIVING_DUTY days

  const weekMon = toDateStr(isoWeekMonday(day.date))
  const driving = day.totalDrivingMinutes
  // … daily driving limit + break checks
```

**Fix:** Split the guard so daily-hours check fires for all working days, but break check only fires for driving days:
```typescript
for (const day of days) {
  if (day.isRestDay) continue       // ← keep: truly idle days get no checks

  // ── Rule 1b: Daily Working Hours Limit (for non-driving working days) ──
  // We only check this when the day has duty time but no driving.
  // When hasDriving is true, the driving-limit check below also contributes.
  if (!day.hasDriving) {
    const dutyMins = day.totalDutyMinutes
    if (dutyMins > MAX_DAILY_WORKING_REDUCED) {
      issues.push({ ruleId: "EU_DAILY_WORK_LIMIT", severity: "violation", ... })
    } else if (dutyMins > MAX_DAILY_WORKING) {
      issues.push({ ruleId: "EU_DAILY_WORK_LIMIT", severity: "warning", ... })
    }
    continue  // no driving = skip the driving-specific checks below
  }

  // Below here only runs when hasDriving is true
  const weekMon = ...
  const driving = day.totalDrivingMinutes
  // Rule 1: Daily Driving Limit ...
  // Rule 2: Breaks ...
```

**Constants needed:** `EU_DAILY_WORK_LIMIT` rule ID, `MAX_DAILY_WORKING = 13 * 60` (780 min), `MAX_DAILY_WORKING_REDUCED = 15 * 60` (900 min). These need to be added to the constants section of `assimilated.ts`.

**Impact:** After fix, a driver with an 14h NON_DRIVING_DUTY trip will correctly get a warning. A driver with a 16h trip gets a violation. Currently neither fires.

### 1. `weekly-rest.test.ts` (NEW FILE)

Tests for `EU_WEEKLY_REST` and `EU_CONSECUTIVE_REDUCED_WEEKLY_REST` rules in `assimilated.ts`.

**Scenario A — 45h rest (compliant):**
- Day 1 Fri: 06:00–18:00 (last activity ends 18:00)
- Day 2 Mon: 15:00–23:00 (next activity starts Mon 15:00)
- Gap: Fri 18:00 → Mon 15:00 = 69h → no weekly rest violation

**Scenario B — reduced weekly rest (24h–45h, first occurrence, no violation):**
- Day 1 Sat: 06:00–22:00 (ends 22:00)
- Day 2 Mon: 08:00–16:00 (starts 34h later)
- Gap = 34h (24h ≤ rest < 45h) → no violation (first reduced rest is allowed)

**Scenario C — weekly rest < 24h (violation):**
- Day 1 Sat: 06:00–22:00
- Day 2 Sun: 18:00–23:00 (starts 20h after previous day ends)
- Gap = 20h < 24h → `EU_WEEKLY_REST` violation

**Scenario D — two consecutive reduced weekly rests (violation):**
- Week 1: includes a 30h gap (reduced rest)
- Week 2: also includes a 30h gap (reduced rest again)
- Expected: `EU_CONSECUTIVE_REDUCED_WEEKLY_REST` violation in Week 2

## `breaksAssumedCompliant` Flag

**Location:** `apps/v4/lib/compliance-engine/types.ts` L167–173

**Add to `VehicleConfig`:**
```typescript
export interface VehicleConfig {
  vehicleType:              VehicleType
  usageType:                UsageType
  vehicleWeightTonnes:      number
  isInternational?:         boolean
  /**
   * When true (default), breaks within working windows are assumed compliant.
   * The EU break requirement check (45min per 4.5h driving) only fires for

  usageType:               UsageType.STANDARD,

}
```

**Wiring in `assimilated.ts`:** The break check (Rule 2, currently at L190–202) is already gated inside `if (!day.hasDriving) continue` — after the D-10 fix, `continue` for non-driving days means the break check naturally only runs for driving days. The `breaksAssumedCompliant` flag is therefore already implicitly honoured. No additional code change needed for the flag wiring; the flag's presence in the type is the forward-compatibility hook.

## Trips List Badge

**File:** `apps/v4/app/(dashboard)/trips/page.tsx`
**Target:** `colDefs` array, "Est. End" column (L1730–1737)

**Current cellRenderer:**
```tsx
cellRenderer: ({ value }: ICellRendererParams) => (
  <span className="text-xs text-muted-foreground">{formatDate(value)}</span>
),
```

**Replacement cellRenderer:**
```tsx
cellRenderer: ({ value, data }: ICellRendererParams<Order>) => {
  const hasEndTime = value || (data?.time && data.time > 0)
  if (!hasEndTime) {
    return (
      <span className="inline-flex items-center gap-1 rounded-md bg-amber-100 px-1.5 py-0.5 text-[10px] font-semibold text-amber-700 dark:bg-amber-900/30 dark:text-amber-400">
        ⚠ No end time
      </span>
    )
  }
  return <span className="text-xs text-muted-foreground">{formatDate(value)}</span>
},
```

The condition: `!value` (estimated_end_date is null/empty) AND `!data?.time || data.time <= 0` (time field also absent/zero) → show badge. Otherwise show the date as normal.
