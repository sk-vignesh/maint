# Plan 02: Harden Prospective Check & Fix Race Condition

<objective>
Fix the race condition where fast consecutive drops bypass overlap detection because `tripIndex` (React state) lags behind synchronous localStorage writes. Also fix the `confirmReassign` path which doesn't write `trip_data` to localStorage.
</objective>

<must_haves>
- `confirmReassign` writes `trip_data` to localStorage just like `handleDrop` does
- Prospective check prioritizes localStorage `trip_data` over async-lagging `tripIndex`
- Two rapid consecutive drops cannot both succeed if they would create an overlap
</must_haves>

## Tasks

<task id="02.1">
<title>Fix confirmReassign to write trip_data</title>
<read_first>
- apps/v4/app/(dashboard)/rota/page.tsx (lines 1134-1182 — confirmReassign function)
- apps/v4/app/(dashboard)/rota/page.tsx (lines 1106-1119 — handleDrop for comparison — notice it writes trip_data)
</read_first>
<action>
In `confirmReassign()` in rota/page.tsx, the `upsertRota()` call at line 1164 does NOT include `trip_data`. Fix it:

Replace lines 1163-1170:
```typescript
    const existing = getEntry(driver.uuid, date)
    upsertRota({
      driver_uuid: driver.uuid,
      date,
      status: "WD",
      trip_uuids: [newTripUuid],
      note: existing?.note,
    })
```

With:
```typescript
    const existing = getEntry(driver.uuid, date)
    upsertRota({
      driver_uuid: driver.uuid,
      date,
      status: "WD",
      trip_uuids: [newTripUuid],
      note: existing?.note,
      // Embed timing data so prospective check doesn't depend on async tripIndex
      trip_data: tripOrder ? [{
        uuid: newTripUuid,
        scheduled_at: tripOrder.scheduled_at ?? undefined,
        estimated_end_date: tripOrder.estimated_end_date ?? undefined,
        time: tripOrder.time,
      }] : undefined,
    })
```

This matches the pattern used in `handleDrop` (lines 1113-1118).
</action>
<acceptance_criteria>
- `confirmReassign` calls `upsertRota` with a `trip_data` field containing the new trip's timing data
- The `trip_data` field matches the same structure used in `handleDrop` (uuid, scheduled_at, estimated_end_date, time)
- After reassignment, the rota entry in localStorage includes embedded timing data
</acceptance_criteria>
</task>

<title>Verify prospective check data source priority</title>
<read_first>

<action>
Review the existing data collection in `prospectiveComplianceCheck()`. The current implementation already:

1. **Source A** (lines 129-135): Reads from `tripIndex` (React state — async, potentially stale)
2. **Source B** (lines 142-167): Reads from localStorage rota entries with `trip_data` and `trip_uuids`

Source B uses `trip_data` (embedded timing data) with priority, falling back to `trip_uuids` resolved via `tripIndex`.

The key insight: `driverTripMap` is built with `has()` checks, so if an entry is already found via Source B (localStorage), it won't be overwritten by Source A. And Source B reads from `trip_data` first (line 145), which is written synchronously.

**No code change needed** — the current priority order is correct. The fix was in task 02.1 (making `confirmReassign` actually WRITE `trip_data`). Without that write, Source B had nothing to read for reassignment cases.

Document this verification in a code comment at line 122:
```typescript
  // ── Step 1: Collect ALL current trips for this driver ──────────────────────
  // Priority: localStorage trip_data (synchronous, always current) > tripIndex (async React state).
  // This ordering is critical for race-condition safety: two rapid drops both write trip_data
  // synchronously before either's async updateOrder() completes, so the second drop's
  // prospective check always sees the first drop's data via localStorage.
```
</action>
